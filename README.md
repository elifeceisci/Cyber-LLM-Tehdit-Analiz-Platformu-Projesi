
Cyber-LLM Threat Analysis Platform Project

1. Project Summary
In today’s cybersecurity ecosystem, thousands of log entries are generated every second. It is inefficient in terms of time and resources for a security analyst to manually review this raw data (IP addresses, port information, error codes) and convert it into a meaningful attack report.
The objective of this project is to develop an AI assistant that automatically analyzes raw system logs using Natural Language Processing (NLP) techniques and Large Language Models (LLM), determines the threat level, and provides an action plan within seconds.

2. Problem Definition
Today, cybersecurity teams are faced with a vast number of logs, incidents, and alerts. Manually analyzing this data leads to wasted time, missed critical threats, and increased operational costs. The goal is to address the lack of an automated classification system capable of quickly and consistently determining the threat level (Low–Medium–Critical) of cyber incident data.

3. Solution Approach
This problem has been addressed using an LLM-based hybrid architecture.
The developed solution consists of two main layers:
1.    LLM-Based Semantic Analysis Layer
    Processes natural language representations of cyber incident/log data.
    Interprets the context, severity, and potential impact of incidents.
    Numeric and categorical features are derived from LLM outputs.
2.    Machine Learning-Based Classification Layer
    Features generated by the LLM are fed into the model as input.
    A threat level (Low – Medium – Critical) prediction is made.

Thanks to this hybrid approach, the LLM’s strong contextual understanding capabilities are combined with the robust structure of traditional classification algorithms.

4.4 Hardware Selection: 
The Cyber-LLM project was implemented using the high-parallelism CUDA cores and Tensor Core support of the NVIDIA T4 GPU, running on the free, installation-free cloud infrastructure provided by Google Colab. 
This hardware choice, which provides a significant speedup over the CPU in vector and matrix-based operations, optimizes model training and evaluation processes while offering an ideal energy-efficiency balance for mid-scale cybersecurity projects.
The platform’s hardware-agnostic development capabilities eliminate the hassle of setup, enabling rapid prototyping in both academic and professional projects. Additionally, this infrastructure already lays the scalable foundation needed to migrate the current system to more complex deep learning architectures, such as LSTM or Transformer, in the future.

4.5 Implementation
4-bit Quantization Method:
BitsAndBytesConfig: This structure implements the 4-bit quantization (compression) method.
    “load_in_4bit=True”: Reduces VRAM usage by 75–80% by downsampling the model’s weights from 32-bit to 4-bit.
    “bnb_4bit_quant_type=’nf4’ ”: “NormalFloat4” is a specialized data type used in technical texts such as cybersecurity, providing higher accuracy than standard 4-bit values.
    “bnb_4bit_compute_dtype=torch.float16”: Uses 16-bit precision (half-precision) to increase speed during computations.

Tokenization and Model Loading
•    AutoTokenizer: Divides the log text in human language into numerical segments that the model can mathematically understand.
•    device_map="auto": Prevents “Out of Memory” errors by efficiently distributing the model’s layers between the graphics card (GPU) and the processor (CPU).

“Chain of Thought” Prompt Engineering
The prompt structure within the code ensures that the model not only makes predictions but also follows a logical analysis process:
•    Method: A role is defined for the model by stating “Analyst Mode: ACTIVE,” and analysis steps (1. Threat Level, 2. Attack Type, 3. Solution) are requested item by item.
•    Result: This method ensures the model generates a report adhering to cybersecurity standards (Reasoning) rather than producing random text.

Analysis Logic and Parameters (model.generate)
•    max_new_tokens=200: Limits the length of the generated response, preventing unnecessary word clutter and improving response speed.
•    temperature=0.3: Determines how “creative” or “precise” the response will be. A low value like 0.3 ensures the generation of serious and consistent responses required in cybersecurity.


Statistical Visualization and Data Processing
The textual report generated by the model is analyzed in the background by the code:
•    Keyword Detection: The `if “critical” in response.lower()` structure scans the text generated by the LLM to identify the attack type and updates the `stats` dictionary.
•    create_stat_plot: Converts these statistics into a visual bar chart using Matplotlib. Thanks to the use of io.BytesIO(), the graph is transferred directly from memory to the interface without taking up disk space.

Gradio Web Interface Layer
This layer enables the user to experience the project as a “product”:
•    gr.Blocks: Sets up a flexible layout (input box on the left, result report and graph on the right).
•    theme=gr.themes.Soft(): Gives the interface the appearance of a professional cybersecurity dashboard.
•    demo.launch(debug=True): Deploys the application and allows the developer to view errors in real time.
The code structure is designed with modularity in mind, enabling future integration with REST APIs, web interfaces, or real-time systems.

Model Performance Summary:
•  Strengths: The model demonstrated 100% precision in all cases it classified as “Critical,” proving that it did not generate any false positives. The model is highly reliable in “accurately identifying the most high-risk attacks,” which is the most critical success metric in cybersecurity.
•  Weaknesses and Confusions: The model struggles most with “Moderate” level threats, sometimes classifying them as ‘Low’ and other times as “Critical.” According to the confusion matrix, the fact that 1 out of 3 critical cases was predicted as “Moderate” indicates that the system sometimes underestimates the severity of the attack.
•  Development Needs: This test, conducted on a total of 6 examples (support), should be supported by a statistically broader dataset. Training the model through fine-tuning tailored to cybersecurity jargon to resolve the uncertainty in medium-level classifications is the top priority for further development.

6. Final Evaluation and Future Work
This project is open to further development with a focus on LLMs. Planned advanced features include:
Hardware Scaling: Accuracy can be improved by transitioning to Llama 3 (70B) models using high-capacity GPUs such as NVIDIA H100 or A100.
Fine-Tuning: By performing “fine-tuning” with model-specific cybersecurity datasets, the model’s success in understanding technical jargon will be enhanced.
RAG Integration: By connecting current CVE (vulnerability) databases to the model, the model will be kept instantly informed of the latest attacks worldwide.
Groq API Integration: By enabling this integration, the model can run much faster on the massive GPUs in the Groq cloud rather than on limited computer hardware.
